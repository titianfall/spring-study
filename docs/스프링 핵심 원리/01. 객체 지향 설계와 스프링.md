# 1. 객체 지향 설계와 스프링

> 강의자료: `객체 지향 설계와 스프링-ppt.pdf`
> 핵심: 스프링은 **좋은 객체 지향 애플리케이션을 개발하도록 도와주는 프레임워크**다. 그 중심에는 **다형성**과 **SOLID(특히 OCP·DIP)**, 그리고 그것을 완성해주는 **DI**가 있다.

> [!NOTE]

## 스프링의 탄생 배경

- **EJB(Enterprise Java Beans)** 시대: 자바 진영의 표준이었으나 너무 복잡하고 무거웠다 (이른바 "EJB 지옥").
- **스프링**: EJB 컨테이너를 대체하며 등장. **단순함의 승리**로 현재 사실상 표준 기술이 되었다.
- **하이버네이트**: EJB 엔티티빈을 대체 → 이후 **JPA(Java Persistence API)** 라는 새 표준을 정의.
  - JPA는 표준 인터페이스, 구현체로 **하이버네이트 / EclipseLink** 등이 있다.

- "스프링"이라는 이름은 J2EE(EJB)라는 **겨울**을 넘은 **새로운 시작(봄)** 이라는 뜻.

## 스프링이란?

### 스프링 생태계

- **필수**: 스프링 프레임워크, 스프링 부트
- **선택**: 스프링 클라우드, 시큐리티, 세션, 배치, REST Docs, 데이터 등

**스프링 프레임워크** 주요 기술:
- 핵심 기술: **DI 컨테이너, AOP, 이벤트**
- 웹 기술: 스프링 MVC, WebFlux
- 데이터 접근: 트랜잭션, JDBC, ORM, XML
- 기술 통합 / 테스트 / 언어(코틀린, 그루비)

**스프링 부트**:
- 스프링을 편리하게 사용하도록 지원, 최근에는 기본으로 사용
- **Tomcat 내장** → 별도 웹 서버 설치 불필요
- **starter** 종속성으로 손쉬운 빌드 구성, 3rd party 라이브러리 자동 구성
- **관례에 의한 간결한 설정**, 프로덕션 준비 기능(메트릭·상태확인)

> [!NOTE]
> "스프링"이란 단어는 문맥에 따라 **DI 컨테이너 기술 / 프레임워크 / 스프링 생태계 전체**를 의미할 수 있다.

---

## 스프링의 진짜 핵심

- 스프링은 **자바 언어 기반**의 프레임워크.
- 자바의 가장 큰 특징은 **객체 지향 언어**라는 점.
- **스프링은 객체 지향 언어의 강력한 특징을 살려, 좋은 객체 지향 애플리케이션을 개발하도록 도와주는 프레임워크다.**

---

## 좋은 객체 지향 프로그래밍

객체 지향 특징: **추상화 · 캡슐화 · 상속 · 다형성**

- 프로그램을 명령어의 목록이 아니라, 여러 개의 독립된 **"객체"들의 모임**으로 본다.
- 객체는 서로 **메시지를 주고받으며 협력**한다.
- 목적: 프로그램을 **유연하고 변경이 용이하게** 만든다. (레고 블럭·컴퓨터 부품 갈아 끼우듯)

### 다형성 (Polymorphism)

**역할과 구현으로 세상을 구분**한다.

| 비유 | 역할 | 구현 |
| --- | --- | --- |
| 운전 | 자동차 역할 | K3, 아반떼, 테슬라 모델3 |
| 공연 | 로미오 / 줄리엣 역할 | 장동건·원빈 / 김태희·송혜교 |

**역할과 구현을 분리했을 때의 장점 (클라이언트 관점):**
- 대상의 **역할(인터페이스)만** 알면 된다.
- 구현 대상의 **내부 구조를 몰라도** 된다.
- 구현 대상의 **내부 구조가 변경되어도** 영향받지 않는다.
- 구현 대상 **자체를 변경해도** 영향받지 않는다.

**자바 언어로의 매핑:**
- 역할 = **인터페이스**
- 구현 = 인터페이스를 구현한 클래스 / 구현 객체
- 설계 시 역할(인터페이스)을 먼저 부여하고, 그 역할을 수행하는 구현 객체를 만든다.

**협력 관계:**
- 혼자 있는 객체는 없다. **클라이언트(요청) ↔ 서버(응답)** 로 서로 협력한다.
- 자바의 **오버라이딩 + 다형성** 덕에, 인터페이스를 구현한 객체를 **실행 시점에 유연하게 변경**할 수 있다.

```java
// 구현 객체만 바꾸면 클라이언트(MemberService)는 그대로
private MemberRepository memberRepository = new MemoryMemberRepository();
// private MemberRepository memberRepository = new JdbcMemberRepository();
```

**다형성의 본질:**
- 인터페이스를 구현한 객체 인스턴스를 **실행 시점에 유연하게 변경**할 수 있다.
- **클라이언트를 변경하지 않고, 서버의 구현 기능을 유연하게 변경**할 수 있다.

> [!WARNING]
> **한계:** 역할(인터페이스) **자체가 변하면** 클라이언트·서버 모두에 큰 변경이 발생한다.
> (자동차→비행기, 대본 변경, USB 인터페이스 변경 등) → **인터페이스를 안정적으로 잘 설계하는 것이 중요.**

## 좋은 객체 지향 설계의 5가지 원칙 (SOLID)

클린코드로 유명한 **로버트 마틴**이 정리.

| 원칙 | 이름 | 핵심 |
| --- | --- | --- |
| **SRP** | 단일 책임 원칙 | 한 클래스는 하나의 책임만. 기준은 **변경** — 변경 시 파급 효과가 적어야 한다. (예: 생성과 사용의 분리) |
| **OCP** | 개방-폐쇄 원칙 | 확장에는 **열려**, 변경에는 **닫혀** 있어야 한다. 다형성으로 새 구현 클래스를 추가해 확장한다. |
| **LSP** | 리스코프 치환 원칙 | 하위 타입은 인터페이스 규약을 지켜 상위 타입을 대체할 수 있어야 한다. (예: 엑셀은 느려도 **앞으로** 가야 함) |
| **ISP** | 인터페이스 분리 원칙 | 범용 인터페이스 하나보다 특정 클라이언트용 여러 개가 낫다. (자동차 → 운전/정비 인터페이스로 분리) |
| **DIP** | 의존관계 역전 원칙 | **추상화에 의존하고 구체화에 의존하지 마라.** 구현 클래스가 아닌 인터페이스(역할)에 의존해야 유연하다. |

### OCP·DIP의 딜레마

```java
public class MemberService {
    // 인터페이스에 의존하지만... new 로 구현 클래스도 동시에 의존한다
    private MemberRepository memberRepository = new MemoryMemberRepository();
}
```

- `MemberService`가 **구현 클래스를 직접 선택** → 구현체를 바꾸려면 **클라이언트 코드를 변경**해야 한다.
- 분명 다형성을 사용했지만 **OCP 위반**이고, 인터페이스와 구현 클래스에 동시에 의존하므로 **DIP 위반**이다.

> [!IMPORTANT]
> **정리: 다형성만으로는 부족하다.**
> 다형성만으로는 구현 객체를 변경할 때 클라이언트 코드도 함께 변경된다 → **OCP·DIP를 지킬 수 없다.**
> 객체를 생성하고 연관관계를 맺어주는 **별도의 조립·설정자**가 필요하다.

## 객체 지향 설계와 스프링

**스프링 이야기에 왜 객체 지향이 나오는가?**

- 스프링은 **DI(Dependency Injection, 의존관계 주입)** 와 **DI 컨테이너**로 다형성 + OCP + DIP를 가능하게 한다.
- 덕분에 **클라이언트 코드 변경 없이 기능을 확장**하고, 부품 교체하듯 개발할 수 있다.
- 스프링의 **IoC(제어의 역전), DI**는 결국 다형성을 활용해 역할과 구현을 편리하게 다루도록 지원하는 것.

> 순수하게 자바로 OCP·DIP를 지키며 개발하다 보면, 결국 **DI 컨테이너(=스프링)** 를 직접 만들게 된다. DI는 말로는 이해가 어렵고 **코드로 짜봐야** 필요성을 알게 된다. → **02장 예제에서 직접 확인한다.**

### 정리

- **모든 설계에 역할과 구현을 분리하자.** (자동차·공연처럼 배역만 만들고 배우는 유연하게 교체)
- 이상적으로는 모든 설계에 인터페이스를 부여하자.

> [!TIP]
> **실무 고민:** 인터페이스를 도입하면 **추상화 비용**이 발생한다. 확장 가능성이 없다면 **구체 클래스를 직접 사용**하고, 꼭 필요할 때 리팩터링으로 인터페이스를 도입하는 것도 방법이다.
