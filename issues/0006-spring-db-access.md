# 6. 스프링 DB 접근 기술

> 강의자료: `6. 스프링 DB 접근 기술.pdf`  
> 현재 프로젝트 기준: Spring Boot `3.5.16` / Java `17` / Gradle Groovy DSL

---

## 요약

- 데이터를 메모리가 아니라 **실제 DB(H2)** 에 저장하도록 바꾼다.
- 같은 `MemberRepository` 인터페이스를 두고 구현 기술만 갈아끼운다:
  **순수 JDBC → JdbcTemplate → JPA → 스프링 데이터 JPA**.
- 핵심은 **스프링 DI + OCP(개방-폐쇄 원칙)**: `SpringConfig`의 빈 등록 한 줄만 바꾸면
  서비스/컨트롤러 코드를 전혀 건드리지 않고 저장 기술을 교체할 수 있다.
- 뒤로 갈수록 **반복 코드가 사라진다**. JDBC는 직접 다 짜고, 스프링 데이터 JPA는 인터페이스만 선언한다.

## H2 데이터베이스 설치

- 개발·테스트용으로 가볍고 편리한 DB, **웹 콘솔 화면**을 제공한다. (https://www.h2database.com)
- H2 버전은 **스프링 부트 버전에 맞춘다.**
  - 부트 2.x → H2 `1.4.200`
  - 부트 3.x → H2 `2.1.214` 이상
- 실행: 윈도우는 `h2.bat` (맥/리눅스는 `chmod 755 h2.sh` 후 `./h2.sh`).

### DB 파일 생성 순서 (중요)

```text
1) 최초 한 번만:   jdbc:h2:~/test      → 홈 폴더에 ~/test.mv.db 파일 생성
2) 이후부터는:     jdbc:h2:tcp://localhost/~/test   ← TCP 모드로 접속
```

> **왜 처음엔 `jdbc:h2:~/test`이고 이후엔 `tcp://`인가?**
> 처음 한 번은 파일을 만들기 위해 직접(임베디드) 접속하고,
> 그 다음부터는 **TCP(서버) 모드**로 붙어야 스프링 앱과 H2 콘솔이 같은 DB를 **동시에** 쓸 수 있다.
> (이유는 아래 [H2 TCP 연결과 파일 락 에러](#h2-tcp-연결과-파일-락-에러) 참고)

### H2가 정상 생성되지 않을 때

- `Database "..." not found, either pre-create it or allow remote database creation ...` 메시지가 뜨는 경우가 있다.
- 해결: H2 종료 후 재시작 → 브라우저 주소창의 **앞부분 임의 숫자(예: `100.1.2.3`)만 `localhost`로 변경** 후 Enter
  (뒤의 세션 값은 절대 바꾸지 않는다) → `jdbc:h2:~/test`로 다시 접속하면 정상 생성된다.

### 테이블 생성 (`sql/ddl.sql`)

```sql
drop table if exists member CASCADE;
create table member
(
    id   bigint generated by default as identity,
    name varchar(255),
    primary key (id)
);
```

### H2의 SQL 문법은 무슨 기준인가? (MySQL?)

> **질문**: H2 DB는 문법이 뭐 기준임? MySQL인가?

**MySQL 기준이 아니다. H2는 기본적으로 표준 SQL(ANSI/ISO SQL)을 따르는 자체 방언(dialect)이다.**

- 기본 모드(`REGULAR`)에서는 **표준 SQL에 가깝게** 동작한다. 굳이 비교하면 PostgreSQL/표준 쪽에 가깝고, **MySQL 전용 문법과는 다르다.**
- 증거가 위 DDL에 있다. **자동 증가 PK**를 쓸 때:
  - H2/표준: `id bigint generated by default as identity` ← 우리가 쓴 것 (ANSI 표준)
  - MySQL 전용: `id bigint auto_increment` ← H2 기본 모드에선 안 통함
- 즉 `auto_increment` 같은 MySQL 고유 문법을 그대로 쓰면 H2 기본 모드에선 에러가 난다.

**호환(Compatibility) 모드 지원**

- H2는 다른 DB를 흉내 내는 **호환 모드**를 제공한다. 접속 URL에 `MODE`를 주면 된다.
  ```text
  jdbc:h2:tcp://localhost/~/test;MODE=MySQL       ← MySQL 문법 흉내
  jdbc:h2:tcp://localhost/~/test;MODE=PostgreSQL  ← PostgreSQL 흉내
  jdbc:h2:tcp://localhost/~/test;MODE=Oracle      ← Oracle 흉내
  ```
- 운영 DB가 MySQL이고 **로컬 테스트만 H2로** 하고 싶을 때 `MODE=MySQL`을 줘서 문법 차이를 줄이는 식으로 쓴다.
- 단, 호환 모드도 **완벽한 동일은 아니다.** 그래서 실무에선 가급적 **테스트도 운영과 같은 DB**로 하거나, JPA처럼 **DB 방언(Dialect)을 추상화**해 주는 기술을 써서 DB 종속을 줄인다.

> 한 줄 기억: **H2 기본 = 표준 SQL(자체 방언).** MySQL이 아니다. 필요하면 `MODE=MySQL`로 흉내만 낼 수 있다.
> (이번 강의의 `generated by default as identity`가 표준 SQL이라는 증거)

## H2 TCP 연결과 파일 락 에러

> **질문**: H2 DB에 TCP 연결을 하지 않으면 파일 락이 걸려서 에러가 나는 것인가?
> 여기에서 에러가 나는 원인과 이유는?

### 한 줄 답

맞다. **TCP(서버) 모드가 아니라 임베디드(파일 직접) 모드로 두 프로세스가 같은 `~/test.mv.db`를 동시에 열려고 해서** 락 충돌이 난다.
TCP 모드로 붙으면 **H2 서버 프로세스 하나만 파일을 잡고**, 앱과 콘솔은 그 서버를 통해 접근하므로 충돌이 사라진다.

### 왜 락이 걸리나 (원인)

H2의 접속 방식은 크게 두 가지다.

| 모드 | URL 예시 | 누가 파일을 여는가 |
| --- | --- | --- |
| **임베디드(Embedded)** | `jdbc:h2:~/test` | 접속한 **그 프로세스(JVM)가 직접** `.mv.db` 파일을 연다 |
| **서버(TCP)** | `jdbc:h2:tcp://localhost/~/test` | **H2 서버 프로세스 1개만** 파일을 열고, 클라이언트는 TCP로 서버에 요청 |

- H2는 파일이 깨지는 것을 막기 위해 **기본적으로 DB 파일에 락(file lock)을 건다.**
  임베디드 모드에서는 파일을 연 프로세스가 그 파일을 **독점**한다.
- 그런데 보통 개발 중에는 **프로세스가 2개**다.
  1. H2 콘솔(웹 화면) — `h2.bat`로 띄운 H2 서버/콘솔
  2. 우리 스프링 부트 애플리케이션
- 두 프로세스가 **모두 임베디드(`jdbc:h2:~/test`)로** 같은 파일을 열려고 하면,
  먼저 연 쪽이 락을 잡고 있으므로 나중에 연 쪽이 **락을 못 잡아 에러**가 난다.

```text
org.h2.jdbc.JdbcSQLNonTransientConnectionException:
Database may be already in use: ... "Locked by another process"
```

### 왜 TCP면 해결되나 (이유)

- TCP 모드(`jdbc:h2:tcp://localhost/~/test`)에서는 **파일을 직접 여는 주체가 H2 서버 프로세스 하나뿐**이다.
- 스프링 앱도, H2 콘솔도 모두 **그 서버에게 TCP로 요청**만 보낸다 → 파일을 직접 열지 않는다.
- 그래서 클라이언트가 여러 개여도 **파일 락을 다투지 않는다.** 서버가 동시 접근을 정리해 준다.

```text
[임베디드 — 충돌]
  스프링 앱  ──직접 연다──▶  ~/test.mv.db  ◀──직접 연다──  H2 콘솔
                          (둘이 같은 파일을 잡으려다 락 에러)

[TCP — 정상]
  스프링 앱  ─TCP─▶  H2 서버 ──연다──▶ ~/test.mv.db
  H2 콘솔   ─TCP─▶  (같은 서버)        (파일은 서버 하나만 연다)
```

### 강의 보충 — "소켓을 통해 여러 곳에서 접근"

강의에서 TCP 모드를 두고 **"소켓을 통해 여러 곳에서 접근할 수 있다"** 고 설명한다. 이게 바로 TCP 모드의 본질이다.

- `jdbc:h2:tcp://localhost/~/test`의 **`tcp`** 는 곧 **소켓 통신**을 의미한다.
  H2 서버가 특정 포트(기본 `9092`)에 **소켓을 열어 두고**, 클라이언트들은 그 포트로 **네트워크 접속**한다.
- 임베디드(`jdbc:h2:~/test`)는 **파일을 직접 여는** 방식이라 한 프로세스만 붙을 수 있지만,
  TCP는 **소켓 서버에 붙는** 방식이라 **스프링 앱·H2 콘솔·DB 툴 등 여러 클라이언트가 동시에** 접속할 수 있다.
- 즉 파일을 두고 다투는 게 아니라, **소켓(서버) 한 곳을 창구로 여럿이 줄 서서 쓰는** 구조다.

```text
        ┌─ 스프링 앱 ─┐
소켓     ├─ H2 콘솔  ─┤  ──TCP(소켓, 9092)──▶  H2 서버  ──▶  ~/test.mv.db
(여럿)   └─ DB 툴    ─┘                         (파일은 서버만 연다)
```

> 정리: **"소켓을 통해 접근" = TCP 서버 모드**. 소켓 서버가 동시 접속을 받아 주기 때문에
> 임베디드 모드의 파일 독점 한계를 넘어 **여러 곳에서 같은 DB를 공유**할 수 있는 것이다.
> (localhost라 같은 PC지만, 같은 원리로 원격 IP에서도 접속 가능하다.)

### 정리

- **에러의 직접 원인**: 같은 DB 파일을 **두 프로세스가 임베디드 모드로 동시에 열어** 파일 락이 충돌.
- **TCP 모드의 역할**: 파일 접근을 **서버 1곳으로 단일화**해서 여러 클라이언트가 안전하게 공유 → 락 충돌 제거.
- 그래서 강의가 "최초 1회만 `jdbc:h2:~/test`, 이후엔 `jdbc:h2:tcp://localhost/~/test`"로 안내하는 것이다.
  앱을 돌리는 동안 H2 콘솔도 같이 보려면 **반드시 TCP 모드**여야 한다.

> 한 줄 기억: **임베디드 = 파일 독점(1프로세스), TCP = 서버가 파일을 들고 여럿이 공유.**
> 콘솔과 앱을 동시에 쓰려면 TCP.

## JDBC 드라이버는 항상 필요한가? (JPA를 써도?)

> **질문**: 자바는 DB와 붙으려면 기본적으로 JDBC 드라이버가 필요한가? JPA나 다른 방식을 써도?

### 한 줄 답

**그렇다. JDBC 드라이버는 항상 필요하다.** JPA·JdbcTemplate·MyBatis·스프링 데이터 JPA 모두
**맨 아래에서는 결국 JDBC를 통해** DB와 통신하기 때문이다. 이들은 JDBC를 **대신하는 게 아니라 위에 얹히는** 기술이다.

### 왜 그런가

- **JDBC(Java Database Connectivity)** 는 자바가 **관계형 DB와 통신하는 표준 API**다.
  자바 표준 라이브러리(`java.sql.*`)에 `Connection`, `PreparedStatement`, `ResultSet` 같은 **규격(인터페이스)** 만 정의돼 있다.
- 그 규격을 **각 DB 회사가 구현한 실제 구현체가 "JDBC 드라이버"** 다.
  H2면 `org.h2.Driver`, MySQL이면 `com.mysql.cj.jdbc.Driver` … DB마다 드라이버가 다르다.
- 그래서 자바 ↔ DB 통신은 **"표준 JDBC API ← 구현 ← DB별 드라이버"** 구조다.
  드라이버가 없으면 자바는 그 DB의 통신 규약(프로토콜)을 모르므로 아예 붙을 수 없다.

### JPA를 써도 JDBC가 깔려 있는 이유

```text
[ 우리 코드 ]   em.persist(member)  /  jdbcTemplate.query(...)  /  findByName(...)
      │
      ▼
[ ORM/헬퍼 ]    JPA(Hibernate) · JdbcTemplate · MyBatis · 스프링 데이터 JPA
      │           (객체↔SQL 변환, 반복 코드 제거 — 편의 계층)
      ▼
[ 표준 API ]    JDBC (java.sql.*)        ← 모두 여기로 모인다
      │
      ▼
[ 드라이버 ]    H2 Driver / MySQL Driver ← DB별 구현체 (필수)
      │
      ▼
[   DB    ]     H2, MySQL, ...
```

- JPA(Hibernate)는 객체를 보고 **SQL을 자동 생성**해 주지만, 그 SQL을 **실제로 실행할 때는 JDBC를 호출**한다.
- 그래서 `build.gradle`에서 JPA를 쓸 때도 **`runtimeOnly 'com.h2database:h2'`(드라이버)는 그대로 남겨 둔다.**
  실제로 강의에서도 jdbc starter는 빼도 되지만 **H2 드라이버는 계속 추가**한다.
- 즉 `spring-boot-starter-data-jpa`가 내부에 **jdbc 관련 라이브러리를 포함**하는 것도 이 때문이다.
  (그래서 별도 jdbc starter는 제거해도 되는 것)

### 정리

- JDBC = 자바의 **DB 통신 표준 API**, 드라이버 = 그 API의 **DB별 실제 구현체** → **항상 필요**.
- JPA·JdbcTemplate·MyBatis는 JDBC를 **편하게 감싼 윗 계층**일 뿐, **JDBC를 대체하지 않는다.**
- 추상화 수준만 올라갈 뿐 **바닥은 언제나 JDBC + 드라이버**다.

> 한 줄 기억: **무슨 기술을 쓰든 자바와 DB 사이 바닥에는 항상 JDBC 드라이버가 깔려 있다.**

## DB와 붙을 때는 클라이언트가 필요한가?

> **질문**: DB와 붙을 때는 클라이언트가 필요한가?

### 한 줄 답

**그렇다. DB 연결은 기본적으로 "클라이언트 ↔ 서버" 구조다.**
DB가 **서버**이고, 거기에 접속하는 쪽(우리 스프링 앱, H2 콘솔, DB 툴 등)이 **클라이언트**다.
그리고 자바 앱을 그 "클라이언트"로 만들어 주는 부품이 바로 앞에서 본 **JDBC 드라이버**다.

### 누가 클라이언트인가

- **서버 = DB**: 데이터를 들고 있고, 요청(SQL)을 받아 처리해 돌려준다.
- **클라이언트 = DB에 접속하는 모든 것**:
  - 우리 **스프링 부트 앱** (JDBC 드라이버를 통해 접속)
  - **H2 콘솔**(웹 화면), DBeaver·IntelliJ DB 같은 **DB 툴**
- 즉 "클라이언트"는 별도로 깔아야 하는 특별한 프로그램이 아니라, **DB에 붙는 쪽을 부르는 역할 이름**이다.
  우리 앱은 **JDBC 드라이버를 품는 순간 DB 클라이언트가 된다.**

```text
[ 클라이언트들 ]                       [ 서버 ]
 스프링 앱 (JDBC 드라이버) ─┐
 H2 콘솔                 ─┼──요청(SQL)──▶   DB
 DB 툴(DBeaver 등)        ─┘  ◀──결과────
```

### 단, 임베디드 모드는 "클라이언트=서버"가 한 몸

여기서 앞의 H2 TCP/임베디드 구분이 다시 등장한다.

| 모드 | 서버 위치 | 클라이언트-서버 관계 |
| --- | --- | --- |
| **TCP(서버) 모드** | 별도 H2 서버 프로세스 | 클라이언트(앱·콘솔)와 서버가 **분리** → 소켓으로 통신 |
| **임베디드 모드** | 앱의 JVM **내부**에서 DB가 돌아감 | 클라이언트와 서버가 **같은 프로세스** → 네트워크 없이 직접 호출 |

- TCP 모드: 앱(클라이언트)과 DB(서버)가 **다른 프로세스** → 소켓(포트 9092)으로 붙는다. 그래서 여러 클라이언트 동시 접속 가능.
- 임베디드 모드: DB가 앱 **JVM 안에서** 함께 돌기 때문에 "클라이언트와 서버가 한 몸"이다. 네트워크가 필요 없는 대신, 그 파일은 그 프로세스가 독점([앞의 파일 락 에러](#h2-tcp-연결과-파일-락-에러) 원인).

### 정리

- DB 접속은 **클라이언트-서버 모델**: DB가 서버, 접속하는 쪽이 클라이언트.
- 우리 앱은 **JDBC 드라이버를 통해 "DB 클라이언트" 역할**을 한다 → 그래서 드라이버가 곧 클라이언트 부품.
- 다만 **임베디드 모드**처럼 DB가 앱 안에서 도는 경우엔 클라이언트와 서버가 **물리적으로 같은 프로세스**일 뿐, 개념은 동일하다.

> 한 줄 기억: **DB는 서버, 우리 앱은 (JDBC 드라이버를 낀) 클라이언트.** TCP면 둘이 분리, 임베디드면 한 몸.

> 주의(강의 멘트): JDBC API로 직접 코딩하는 건 20년 전 방식이다. **"이렇게 고생했구나" 정도로 참고만** 하고 넘어가자.

`build.gradle`:


`resources/application.properties`:

```properties

```

> 값 뒤에 **공백이 들어가도 같은 에러**가 나니 공백을 모두 제거할 것.

### JDBC 리포지토리의 특징

- `Connection` → `PreparedStatement` → `ResultSet`을 직접 열고, **`finally`에서 일일이 close** 해야 한다.
- 커넥션은 `DataSourceUtils.getConnection(dataSource)`로 얻고 `releaseConnection`으로 반납한다.
  (트랜잭션 동기화를 위해 직접 `dataSource.getConnection()`을 쓰지 않는다.)
- 메서드 하나(save/findById/findAll/findByName)마다 **똑같은 try-catch-finally 보일러플레이트**가 반복된다 → 다음 단계(JdbcTemplate)가 이 반복을 없애 준다.

```java
public Member save(Member member) {
    String sql = "insert into member(name) values(?)";
    Connection conn = null;
    PreparedStatement pstmt = null;
    ResultSet rs = null;
    try {
        conn = getConnection();
        pstmt = conn.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS);
        pstmt.setString(1, member.getName());
        pstmt.executeUpdate();
        rs = pstmt.getGeneratedKeys();
        if (rs.next()) member.setId(rs.getLong(1));
        else throw new SQLException("id 조회 실패");
        return member;
    } catch (Exception e) {
        throw new IllegalStateException(e);
    } finally {
        close(conn, pstmt, rs);   // 반복되는 정리 코드
    }
}
```

### 스프링 설정 변경 (구현 교체 지점)

```java
@Configuration
public class SpringConfig {
    private final DataSource dataSource;

    public SpringConfig(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @Bean
    public MemberService memberService() {
        return new MemberService(memberRepository());
    }

    @Bean
    public MemberRepository memberRepository() {
//      return new MemoryMemberRepository();        // 기존
        return new JdbcMemberRepository(dataSource); // 교체
    }
}
```

- `DataSource`는 **DB 커넥션을 얻을 때 쓰는 객체**. 스프링 부트가 `application.properties`의 연결 정보로
  `DataSource`를 만들어 **스프링 빈으로 등록**해 두기 때문에 생성자로 **DI** 받을 수 있다.

### 개방-폐쇄 원칙(OCP)

- **확장에는 열려 있고, 수정에는 닫혀 있다.**
- 스프링 DI 덕분에 `MemberService`/`MemberController` **코드는 한 줄도 안 고치고**,
  `SpringConfig`의 빈 등록만 바꿔 구현체(`Memory→Jdbc`)를 교체했다.

```text
            interface
MemberService ───▶ MemberRepository
                      ▲          ▲
                      │          │
          MemoryMemberRepository  JdbcMemberRepository
```

### SOLID — 객체지향 설계 5원칙 (OCP의 출처)

> **질문**: SOLID란?

위에서 본 OCP는 **SOLID**라는 5가지 객체지향 설계 원칙 중 하나다. 좋은(유연하고 유지보수 쉬운) 구조를 위한 가이드라인이며, 다섯 원칙의 앞 글자를 따서 SOLID라고 부른다.

| 약자 | 원칙 | 한 줄 의미 |
| --- | --- | --- |
| **S** | SRP (단일 책임 원칙) | 클래스는 **하나의 책임**만 가져야 한다. 바뀌는 이유가 하나뿐이어야 함 |
| **O** | OCP (개방-폐쇄 원칙) | **확장에는 열고, 수정에는 닫는다.** 기존 코드 수정 없이 기능 확장 |
| **L** | LSP (리스코프 치환 원칙) | 자식(구현체)은 부모(인터페이스)를 **완벽히 대체**할 수 있어야 한다 |
| **I** | ISP (인터페이스 분리 원칙) | **안 쓰는 메서드에 의존하지 않게** 인터페이스를 잘게 나눈다 |
| **D** | DIP (의존관계 역전 원칙) | 구현 클래스가 아니라 **추상(인터페이스)에 의존**한다 |

### 이번 장 코드에 비춰 보기

이번 회원 리포지토리 교체 과정은 SOLID 중 특히 **OCP·DIP·LSP**가 그대로 드러난다.

- **DIP**: `MemberService`가 `JdbcMemberRepository`(구현)가 아니라 **`MemberRepository`(인터페이스)에 의존**한다.
  ```java
  public class MemberService {
      private final MemberRepository memberRepository; // 구현체가 아닌 인터페이스에 의존 (DIP)
  }
  ```
- **OCP**: 그래서 `SpringConfig` 빈 등록만 `Memory→Jdbc→JdbcTemplate→JPA`로 바꿔도
  **서비스 코드는 수정 없이 기능이 확장**된다.
- **LSP**: `MemoryMemberRepository`든 `JdbcMemberRepository`든 **`MemberRepository` 자리에 그대로 갈아끼워도** 동작이 보장된다.
- **이걸 가능케 하는 게 스프링 DI**: 추상에 의존하도록 만들고, 실제 구현체는 컨테이너가 주입해 준다.

> 한 줄 기억: **SOLID = 좋은 객체지향 설계의 5원칙.** 이번 장의 "인터페이스 + DI로 구현 교체"는
> 그중 **DIP(추상에 의존) → OCP(수정 없이 확장)** 가 실제로 동작하는 장면이다.

## 테스트는 자바 메모리에서 끝나는가?

> **질문**: 테스트는 자바 메모리에서 끝나는가?

### 한 줄 답

**테스트 종류에 따라 다르다.** 두 가지를 구분해야 한다.

| 구분 | 단위 테스트 (이전 장) | 통합 테스트 (이번 장) |
| --- | --- | --- |
| 예시 | `MemoryMemberRepository`로 직접 `new` 해서 테스트 | `@SpringBootTest` + 실제 DB |
| 저장소 | 자바 **메모리(HashMap)** | **실제 H2 DB** |
| 스프링 컨테이너 | **안 띄움** | **띄움** (빈 주입) |
| DB 접근 | **없음 → JVM 메모리 안에서 끝남** | **있음 → DB까지 다녀옴** |
| 속도 | 매우 빠름 | 상대적으로 느림 |

- **단위 테스트**: `MemoryMemberRepository`는 데이터를 `Map`에 담는다. 테스트가 끝나고 JVM이 내려가면 **메모리도 같이 사라진다.** → **자바 메모리에서 끝난다.** (DB·네트워크 전혀 관여 안 함)
- **통합 테스트(`@SpringBootTest`)**: 스프링 컨테이너를 띄우고 **실제 H2 DB까지 연결**해 검증한다. → **메모리에서 끝나지 않고 DB까지 실제로 갔다 온다.**

### 통합 테스트인데 DB에 데이터가 안 남는 이유 (헷갈림 주의)

통합 테스트는 DB까지 가지만, `@Transactional` 때문에 **결과가 DB에 남지는 않는다.**

```text
[통합 테스트 1건의 흐름]
 트랜잭션 시작 → INSERT/SELECT 실제 실행(DB 반영됨) → 테스트 종료 → 롤백(되돌림)
                                                          └ commit 안 함
```

- 즉 "DB에 갔다 왔지만, 마지막에 롤백해서 흔적을 지운다." → DB에 **데이터가 안 남는 것**이지,
  **메모리에서만 끝난 것은 아니다.** (실제로 SQL은 DB에서 실행됨)
- 이걸 "메모리에서 끝났다"와 혼동하기 쉽다. **롤백 ≠ 메모리 전용.**

### 정리

- **단위 테스트** = 자바 메모리에서 끝난다 (DB 없음).
- **통합 테스트** = DB까지 실제로 다녀온다. 다만 `@Transactional` 롤백으로 **데이터를 남기지 않을** 뿐.

> 한 줄 기억: **메모리에서 끝나는 건 단위 테스트.** 통합 테스트는 DB까지 가되, 롤백으로 치운다.

## 스프링 통합 테스트

- 단위 테스트와 달리 **스프링 컨테이너 + 실제 DB까지 띄워서** 검증한다.

```java
@SpringBootTest
@Transactional
class MemberServiceIntegrationTest {

    @Autowired MemberService memberService;
    @Autowired MemberRepository memberRepository;

    @Test
    void 회원가입() {
        Member member = new Member();
        member.setName("hello");

        Long saveId = memberService.join(member);

        Member findMember = memberRepository.findById(saveId).get();
        assertEquals(member.getName(), findMember.getName());
    }
}
```

| 애노테이션 | 의미 |
| --- | --- |
| `@SpringBootTest` | **스프링 컨테이너와 함께** 테스트 실행 (실제 빈 주입) |
| `@Transactional` | 테스트 시작 시 트랜잭션 시작, **끝나면 항상 롤백** → DB에 데이터가 남지 않아 다음 테스트에 영향 없음 |

> `@Transactional`이 테스트에 붙으면 **커밋하지 않고 롤백**하는 게 핵심.
> 그래서 같은 테스트를 반복 실행해도 "이미 존재하는 회원" 같은 충돌이 안 난다.

### `assertEquals`는 왜 JUnit 걸 썼나? (assertThat과 섞인 이유)

> **질문**: `assertEquals(member.getName(), findMember.getName())` 여기선 왜 JUnit 걸 씀?

이 통합 테스트의 import를 보면 **두 라이브러리가 섞여** 있다.

```java
import static org.assertj.core.api.Assertions.assertThat;     // AssertJ
import static org.junit.jupiter.api.Assertions.assertEquals;  // JUnit
import static org.junit.jupiter.api.Assertions.assertThrows;  // JUnit
```

- `회원가입()` 테스트 → **JUnit**의 `assertEquals(...)`
- `중복_회원_예외()` 테스트 → **AssertJ**의 `assertThat(...).isEqualTo(...)`

**왜 섞였나?** 깊은 기술적 이유는 없다. **강의 코드가 두 스타일을 다 보여주려고(혹은 그냥 일관성 없이) 섞어 쓴 것**이다.
둘 다 "값이 같은지" 검증하는 같은 목적이고, **기능상 차이 없이 잘 동작**한다.

| | JUnit `assertEquals` | AssertJ `assertThat` |
| --- | --- | --- |
| 호출 형태 | `assertEquals(expected, actual)` | `assertThat(actual).isEqualTo(expected)` |
| 인자 순서 | **기대값이 앞**, 실제값이 뒤 (헷갈리기 쉬움) | 실제값을 먼저 쓰고 **읽듯이** 체이닝 |
| 출신 | JUnit Jupiter (`org.junit.jupiter.api`) | AssertJ (`org.assertj.core.api`) |
| 가독성/메시지 | 평이함 | **자연어처럼 읽히고 실패 메시지가 친절**, 메서드 체인 풍부 |

**실무/강의 권장은 AssertJ(`assertThat`)** 다. 가독성과 실패 메시지가 좋고 인자 순서 혼동이 없어서다.
그래서 이 `assertEquals`도 일관성 있게 바꾸려면 이렇게 쓸 수 있다:

```java
// 기존 (JUnit)
assertEquals(member.getName(), findMember.getName());

// 권장 (AssertJ) — 둘 다 동일하게 동작
assertThat(findMember.getName()).isEqualTo(member.getName());
```

> 한 줄 기억: **둘은 같은 일을 하는 다른 라이브러리.** 강의가 섞어 썼을 뿐이고,
> 보통은 **AssertJ `assertThat`으로 통일**하는 게 권장된다. (`assertThrows`는 JUnit이 담당)

### 트러블슈팅 — "DB에 데이터가 있는데 왜 중복 예외가 안 터지나"

> **상황**: H2 콘솔에서 `SELECT * FROM MEMBER`를 보니 `spring`, `spring2`, `jpa`가 실제로 들어 있다.
> "데이터가 DB에 있으니 가입하면 중복 예외가 터져야 하는 것 아닌가?"

**중복 예외는 "가입하려는 이름"이 DB에 있을 때만 터진다.** 데이터가 DB에 있다는 사실만으로 터지는 게 아니다.

```java
// 회원가입() 테스트
member.setName("hello");                 // 가입 시도 이름 = "hello"
memberService.join(member);

// 중복 검사 로직
memberRepository.findByName(member.getName())   // findByName("hello")
        .ifPresent(m -> { throw new IllegalStateException("이미 존재하는 회원입니다."); });
```

- DB에 있는 이름은 `spring / spring2 / jpa` → **"hello"는 없다.**
- 따라서 `findByName("hello")`는 빈 결과 → `ifPresent`가 실행 안 됨 → **예외 안 터짐(정상)**.
- 터지게 하려면 **이미 있는 이름**으로 가입해야 한다: `member.setName("spring")` → 그때 "이미 존재하는 회원입니다." 발생.

**그런데 이 데이터(`spring/spring2/jpa`)는 어디서 왔나?**
통합 테스트는 `@Transactional`이라 **항상 롤백**되므로 테스트가 남긴 게 아니다.
→ **웹 앱을 직접 실행해 가입**했거나, `@Transactional` 붙기 전 테스트, 또는 H2 콘솔 수동 insert로 **커밋되어 남은 데이터**다.

**이게 바로 `@Transactional`(롤백)이 필요한 이유다.**
만약 `spring2`가 DB에 커밋되어 남아 있으면, `중복_회원_예약` 테스트(`member2="spring2"`)를 돌릴 때
`join("spring2")`가 **그 남은 데이터를 발견해서** `member1`과 무관하게 **엉뚱한 이유로** 예외를 던질 수 있다.
이렇게 남은 데이터가 테스트를 오염시키므로, 매 테스트를 롤백해 **DB를 깨끗한 상태로 되돌리는** 것이다.

> 한 줄 기억: **중복 예외 = "가입하는 이름"이 이미 있을 때만.** 그리고 커밋되어 남은 데이터는
> 테스트를 오염시키므로 `@Transactional`로 롤백해 비운다.

## 스프링 JdbcTemplate

- 환경 설정은 **순수 JDBC와 동일**.
- `JdbcTemplate`/MyBatis는 **JDBC의 반복 코드(연결·예외·자원정리)를 대부분 제거**해 준다.
  단, **SQL은 여전히 직접 작성**해야 한다.

```java
public class JdbcTemplateMemberRepository implements MemberRepository {
    private final JdbcTemplate jdbcTemplate;

    public JdbcTemplateMemberRepository(DataSource dataSource) {
        jdbcTemplate = new JdbcTemplate(dataSource);
    }

    @Override
    public Member save(Member member) {
        SimpleJdbcInsert jdbcInsert = new SimpleJdbcInsert(jdbcTemplate);
        jdbcInsert.withTableName("member").usingGeneratedKeyColumns("id");

        Map<String, Object> parameters = new HashMap<>();
        parameters.put("name", member.getName());

        Number key = jdbcInsert.executeAndReturnKey(new MapSqlParameterSource(parameters));
        member.setId(key.longValue());
        return member;
    }

    @Override
    public Optional<Member> findById(Long id) {
        List<Member> result = jdbcTemplate.query(
            "select * from member where id = ?", memberRowMapper(), id);
        return result.stream().findAny();
    }

    private RowMapper<Member> memberRowMapper() {
        return (rs, rowNum) -> {
            Member member = new Member();
            member.setId(rs.getLong("id"));
            member.setName(rs.getString("name"));
            return member;
        };
    }
}
```

- `RowMapper`: **ResultSet의 한 행 → 객체**로 변환하는 콜백. `Connection`/`finally`가 사라진 게 포인트.
- 설정은 또 한 줄만 교체: `return new JdbcTemplateMemberRepository(dataSource);`

#### 트러블슈팅 — `Column "id" not found` (RowMapper와 SELECT 컬럼 불일치)

> **상황**: `findById`는 잘 되는데 `findByName`만 `JdbcSQLSyntaxErrorException: Column "id" not found`로 터짐.

원인: `findByName`의 SQL만 `select *`가 아니라 **`select name`** 으로 돼 있었다.

```java
// ❌ name만 조회
jdbcTemplate.query("select name from member where name = ?", memberRowMapper(), name);

private RowMapper<Member> memberRowMapper() {
    return (rs, rowNum) -> {
        member.setId(rs.getLong("id"));    // ← 결과에 id 컬럼이 없어서 "Column id not found"
        member.setName(rs.getString("name"));
        ...
    };
}
```

- `RowMapper`가 `rs.getLong("id")`로 **id 컬럼을 읽는데**, `select name`은 name만 가져오므로 ResultSet에 id가 없다 → 에러.
- 해결: SQL을 **`select *`** 로 바꿔 RowMapper가 읽는 컬럼(id, name)을 모두 포함시킨다.

> 한 줄 기억: **RowMapper가 읽는 컬럼 = SELECT가 가져오는 컬럼**이 일치해야 한다.
> (`findById`/`findAll`이 멀쩡한데 한 메서드만 터지면 그 메서드의 SQL 컬럼부터 의심)

### `result.stream().findAny()`란?

> **질문**: `stream().findAny()` 설명 좀.

`jdbcTemplate.query(...)`나 JPA `getResultList()`는 결과를 **항상 `List`로** 돌려준다.
하지만 `findById`/`findByName`의 반환 타입은 **`Optional<Member>`** 다. 이 **`List` → `Optional` 변환**을 해 주는 게 `stream().findAny()`다.

```java
List<Member> result = jdbcTemplate.query("select * from member where id = ?", memberRowMapper(), id);
return result.stream().findAny();   // List<Member> → Optional<Member>
```

| 조각 | 의미 |
| --- | --- |
| `result.stream()` | `List`를 스트림(데이터 흐름)으로 변환 |
| `.findAny()` | 스트림에서 **아무 원소 하나**를 꺼내 **`Optional`로 감싸 반환**. 비어 있으면 `Optional.empty()` |

- **결과가 1개면** 그 원소를 `Optional.of(member)`로, **결과가 0개면** `Optional.empty()`로 → `Optional`이 빈/참 여부를 안전하게 표현.
- 그래서 호출하는 쪽에서 `null` 체크 없이 `.get()`, `.ifPresent()`, `.orElse()` 등을 쓸 수 있다.

**`findAny` vs `findFirst`**

| | `findFirst()` | `findAny()` |
| --- | --- | --- |
| 반환 | 항상 **첫 번째** 원소 | **아무거나 하나** (순차 스트림이면 보통 첫 번째) |
| 병렬 스트림 | 순서 보장 위해 더 느릴 수 있음 | 순서 신경 안 써 **더 빠를 수 있음** |

- 여기서는 `id`/`name`으로 조회해 결과가 **0개 또는 1개**뿐이라 **둘 중 무엇을 써도 결과가 같다.**
- 굳이 순서가 의미 없으니 강의는 `findAny()`를 썼다. (성능상 살짝 유리한 관용적 선택)

> 한 줄 기억: **`stream().findAny()` = "List에서 하나 꺼내 Optional로 포장".**
> 0개면 `empty`, 1개면 그 값. `Optional` 반환 타입을 맞추려는 변환이다.

## JPA

- JPA는 **반복 코드는 물론 기본적인 SQL까지 직접 만들어 실행**해 준다.
- **SQL·데이터 중심 → 객체 중심** 설계로 패러다임 전환, 개발 생산성↑.

`build.gradle` (jdbc 제거, data-jpa 추가):

```groovy
// implementation 'org.springframework.boot:spring-boot-starter-jdbc'
implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
runtimeOnly 'com.h2database:h2'
```

> `spring-boot-starter-data-jpa`는 내부에 **jdbc 라이브러리를 포함**하므로 jdbc starter는 제거해도 된다.

`application.properties` 추가:

```properties
spring.jpa.show-sql=true            # JPA가 생성하는 SQL을 콘솔에 출력
spring.jpa.hibernate.ddl-auto=none  # 테이블 자동생성 끔 (create면 엔티티 기준 테이블 자동생성)
```

### 엔티티 매핑

```java
@Entity
public class Member {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    // getter / setter
}
```

> **스프링 부트 3.0 이상은 `javax.persistence` → `jakarta.persistence`로 반드시 변경.**
> 예: `javax.persistence.Entity` → `jakarta.persistence.Entity`
> (단, `javax.sql.DataSource`는 **그대로 유지** — `javax.sql`은 jakarta로 안 바뀐다.)

#### `GenerationType.IDENTITY`가 오라클 시퀀스인가?

> **질문**: `@GeneratedValue(strategy = GenerationType.IDENTITY)`가 오라클의 시퀀스 타입인가?

**아니다. IDENTITY는 시퀀스가 아니라 "DB의 자동 증가(auto increment) 컬럼"에 PK 생성을 위임하는 방식이다.**
오라클 시퀀스 방식은 **별도의 `GenerationType.SEQUENCE`** 다.

`@GeneratedValue`의 `strategy`는 PK를 **누가/어떻게 만드는지**를 정한다.

| 전략 | 누가 PK를 만드나 | 대표 DB | 비고 |
| --- | --- | --- | --- |
| **IDENTITY** | **DB의 auto increment 컬럼** | MySQL `AUTO_INCREMENT`, **H2 identity**, SQL Server | INSERT 시점에 DB가 채번 → ID를 알려면 **INSERT를 즉시 실행** |
| **SEQUENCE** | **DB의 시퀀스 오브젝트** | **Oracle**, PostgreSQL | `@SequenceGenerator`로 시퀀스 지정. 미리 번호를 받아옴 |
| **TABLE** | 키 생성 전용 테이블 | 모든 DB | 어디서나 되지만 **성능이 떨어짐**, 잘 안 씀 |
| **AUTO**(기본값) | DB 방언에 따라 자동 선택 | - | 명시 안 하면 이게 적용됨 |

- 이번 강의의 DDL이 `id bigint generated by default as **identity**`였다 → **H2의 identity 컬럼** → 그래서 엔티티도 `GenerationType.IDENTITY`로 맞춘 것이다.
- 만약 오라클의 시퀀스를 쓰려면 이렇게 쓴다:
  ```java
  @Id
  @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "member_seq")
  @SequenceGenerator(name = "member_seq", sequenceName = "MEMBER_SEQ", allocationSize = 1)
  private Long id;
  ```

> 한 줄 기억: **IDENTITY = DB 자동증가 컬럼(MySQL/H2), SEQUENCE = 오라클·PostgreSQL 시퀀스.**
> 둘은 다른 전략이다. 우리 DDL이 `identity`라서 IDENTITY를 쓴 것.

### JPA 리포지토리 & 트랜잭션

```java
public class JpaMemberRepository implements MemberRepository {
    private final EntityManager em;

    public JpaMemberRepository(EntityManager em) { this.em = em; }

    public Member save(Member member) {
        em.persist(member);          // INSERT를 JPA가 생성
        return member;
    }
    public Optional<Member> findById(Long id) {
        return Optional.ofNullable(em.find(Member.class, id)); // PK 조회
    }
    public List<Member> findAll() {
        return em.createQuery("select m from Member m", Member.class).getResultList(); // JPQL
    }
    public Optional<Member> findByName(String name) {
        return em.createQuery("select m from Member m where m.name = :name", Member.class)
                 .setParameter("name", name)
                 .getResultList().stream().findAny();
    }
}
```

- `em.find(엔티티, PK)`는 SQL 없이 PK 조회, 그 외에는 **JPQL**(테이블이 아니라 **엔티티(객체)** 를 대상으로 쓰는 쿼리).

#### `EntityManager`는 어떻게 들어오나 (주입 이해)

```java
private final EntityManager em;          // JPA의 핵심 객체를 들고 있음

public JpaMemberRepository(EntityManager em) {
    this.em = em;                        // 스프링이 만들어둔 EntityManager를 주입받음
}
```

- `spring-boot-starter-data-jpa`를 추가하면 **스프링 부트가 JPA 환경을 자동 설정**한다.
  → `DataSource`(DB 연결)를 바탕으로 **`EntityManager`를 만들어 스프링 빈으로 등록**해 둔다.
- 그래서 우리는 **직접 `new` 하지 않고** 생성자로 **주입(DI)** 만 받는다. (이전 `DataSource`/`JdbcTemplate` 받던 DI 패턴과 동일)
- `EntityManager`는 단순 도구가 아니라 **JPA의 모든 DB 작업을 담당하는 핵심 객체**다.
  내부적으로 **영속성 컨텍스트**(persistence context)로 객체를 추적하고, 실제 실행은 **바닥의 JDBC**로 한다.
  ```java
  em.persist(member);              // INSERT
  em.find(Member.class, id);       // PK SELECT
  em.createQuery("select m from Member m", Member.class);  // JPQL
  ```

> 한 줄 기억: **부트가 DataSource로 EntityManager를 만들어 빈 등록 → 우리는 주입받아 꺼내 persist/find/JPQL로 DB 작업.**

##### `@PersistenceContext`는 왜 안 쓰나?

`EntityManager`를 받는 방법은 두 가지인데, **둘 다 "주입(DI)"** 이다. ("DI 없이"가 아니라 방식 차이)

```java
// 방법 1) @PersistenceContext — JPA 표준, 필드 주입
@PersistenceContext
private EntityManager em;

// 방법 2) 생성자 주입 — 스프링 방식 (강의가 쓰는 것)
private final EntityManager em;
public JpaMemberRepository(EntityManager em) { this.em = em; }
```

| | `@PersistenceContext` | 생성자 주입 |
| --- | --- | --- |
| 출신 | **JPA 표준**(스프링 아니어도 동작) | **스프링** DI |
| 위치 | 필드 | 생성자 |
| `final` | ❌ 불가 | ✅ **불변 보장** |

강의가 생성자 주입을 쓰는 이유:

1. **일관성** — `DataSource`/`JdbcTemplate`도 전부 생성자 주입으로 받았으니 동일하게 통일.
2. **스프링 권장** — `final`로 불변, 테스트 쉬움, 의존성이 생성자에 명확히 드러남.
3. **과거엔 `@PersistenceContext`만 가능**했지만, 스프링 부트가 EntityManager도 **생성자 주입을 지원**하면서 이제 어노테이션 없이 받아도 된다.

> 한 줄 기억: **`@PersistenceContext`(JPA 표준 필드주입) 대신 스프링 권장 "생성자 주입"으로 통일한 것.** 둘 다 주입이고 결과는 같다.

- **서비스 계층에 `@Transactional` 필수**:

```java
import org.springframework.transaction.annotation.Transactional;

@Transactional
public class MemberService { }
```

> 스프링은 해당 클래스 메서드 실행 시 **트랜잭션 시작 → 정상 종료면 커밋, 런타임 예외면 롤백**.
> **JPA를 통한 모든 데이터 변경은 트랜잭션 안에서** 실행해야 한다.

설정 교체: `return new JpaMemberRepository(em);` (`EntityManager`도 DI 받음)

#### 트러블슈팅 — JPA 전환 때 실제로 밟은 지뢰 2개

**(1) `@Bean` 메서드에 `@Autowired`를 붙임 → 컨텍스트 로딩 실패**

```text
Configuration problem: @Bean method 'memberServie' must not be declared as autowired;
remove the method-level @Autowired annotation.
```

- 원인: `SpringConfig`에서 `@Bean` 메서드 위에 `@Autowired`가 떠 있었다.
- **`@Bean` 메서드에는 `@Autowired`를 붙이지 않는다.** 의존성은 **설정 클래스 생성자**(또는 메서드 파라미터)로 받는다.
  ```java
  @Autowired
  public SpringConfig(EntityManager em) { this.em = em; }  // 생성자에만 붙임 (OK)

  @Bean                       // ← 여기엔 @Autowired 금지
  public MemberService memberService() { ... }
  ```

**(2) JPQL 명명 파라미터에서 콜론(`:`) 누락 → `No parameter named ':name'`**

```text
org.hibernate.query.UnknownParameterException: No parameter named ':name' in query with named parameters []
```

```java
// ❌ 콜론 없음 → name이 파라미터가 아닌 일반 식으로 해석됨
em.createQuery("select m from Member m where m.name = name", Member.class)
  .setParameter("name", name);

// ✅ :name (명명 파라미터)
em.createQuery("select m from Member m where m.name = :name", Member.class)
  .setParameter("name", name);
```

- JPQL의 **명명 파라미터는 `:이름`** 형태다. 콜론이 빠지면 쿼리에 파라미터가 0개가 되는데
  `setParameter("name", ...)`로 `:name`을 채우려 하니 "그런 파라미터 없다" 에러가 난다.
- `setParameter("name", ...)`의 이름과 **JPQL의 `:name`이 짝이 맞아야** 한다.

> 한 줄 기억: **`@Bean`엔 `@Autowired` 금지(생성자로), JPQL 파라미터는 `:이름`.**

## 스프링 데이터 JPA

- JPA 위에 얹는 프레임워크. **구현 클래스 없이 인터페이스만 선언**하면 기본 CRUD가 다 제공된다.
- **주의**: 스프링 데이터 JPA는 JPA를 편하게 쓰게 도와주는 기술 → **JPA를 먼저 이해한 뒤** 학습할 것.

```java
public interface SpringDataJpaMemberRepository
        extends JpaRepository<Member, Long>, MemberRepository {

    @Override
    Optional<Member> findByName(String name);  // 메서드 이름만으로 쿼리 자동 생성
}
```

설정 — 이제 빈을 `new` 하지 않는다. 스프링 데이터 JPA가 **구현체를 자동 생성·등록**해 주므로 **주입만** 받는다:

```java
@Configuration
public class SpringConfig {
    private final MemberRepository memberRepository;

    public SpringConfig(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;   // 자동 등록된 구현체가 주입됨
    }

    @Bean
    public MemberService memberService() {
        return new MemberService(memberRepository);
    }
}
```

### 제공 기능 / 동작 원리

```text
스프링 데이터:   Repository ◀ CrudRepository ◀ PagingAndSortingRepository
스프링 데이터 JPA:                                      ◀ JpaRepository
```

- 인터페이스 상속만으로 **기본 CRUD**(`save`, `findById`, `findAll`, `count`, `delete` …) 제공.
- `findByName()`, `findByEmail()`처럼 **메서드 이름만으로 조회 쿼리 자동 생성**.
- **페이징** 기능 자동 제공.

> 실무: JPA + 스프링 데이터 JPA를 기본으로 쓰고, **복잡한 동적 쿼리는 Querydsl**,
> 그래도 어려우면 **네이티브 쿼리 / JdbcTemplate**으로 해결.

## 기술 발전 흐름 한눈에

| 단계 | 직접 짜는 것 | 사라진 것 | 설정 변경 |
| --- | --- | --- | --- |
| 순수 JDBC | Connection/SQL/매핑/자원정리 전부 | — | `new JdbcMemberRepository(dataSource)` |
| JdbcTemplate | SQL + RowMapper 매핑 | 연결·예외·자원정리 | `new JdbcTemplateMemberRepository(dataSource)` |
| JPA | 엔티티 매핑 + 일부 JPQL | 기본 SQL | `new JpaMemberRepository(em)` |
| 스프링 데이터 JPA | **인터페이스 선언만** | 구현 클래스 자체 | 빈 자동 등록(주입만) |

> 관통하는 원리: **인터페이스(`MemberRepository`) + DI**.
> 구현이 무엇이든 서비스/컨트롤러는 그대로 → **OCP**가 매 단계 적용된다.

### 현업에서는 대부분 뭘 쓰나?

> **질문**: 현업에서는 JDBC / JdbcTemplate / JPA 중 대부분 뭘 쓰나?

| 기술 | 실무 사용도 | 비고 |
| --- | --- | --- |
| **순수 JDBC** | ❌ 거의 안 씀(직접은) | 반복 코드가 너무 많음. 학습용·원리 이해용 |
| **JdbcTemplate / MyBatis** | 🔸 여전히 많이 쓰임 | **복잡한 SQL·튜닝·통계 쿼리**에 강점. 한국 SI·금융권은 **MyBatis** 강세 |
| **JPA + 스프링 데이터 JPA** | ✅ **주류(대세)** | 신규 프로젝트 기본 선택. 생산성 압도적 |

**현실적인 정리(강의 결론과 동일):**

- **기본은 JPA + 스프링 데이터 JPA.** 실무에서 관계형 DB를 쓴다면 **사실상 필수**다.
  반복 CRUD가 사라지고 핵심 비즈니스 로직에 집중할 수 있다.
- **복잡한 동적 쿼리는 Querydsl**을 곁들인다. 쿼리를 **자바 코드로 타입 안전하게** 작성 → 컴파일 시점에 오류 발견.
- **그래도 안 되는 고난도 쿼리/튜닝**은 JPA의 **네이티브 쿼리**나 **JdbcTemplate**으로 푼다.
- 즉 **"JPA(+스프링 데이터 JPA) + Querydsl 기본, 어려운 건 JdbcTemplate으로 보조"** 조합이 가장 흔하다.

> 참고: 한국 실무에선 **MyBatis**(JdbcTemplate과 비슷하게 SQL을 직접 쓰는 방식)도 SI·금융·레거시에서 여전히 많이 보인다.
> 신규·스타트업·서비스 회사는 **JPA가 기본값**인 경우가 많다.

> 한 줄 기억: **현업 기본은 JPA + 스프링 데이터 JPA**, 복잡 쿼리는 **Querydsl**, 막히면 **JdbcTemplate/네이티브**.
> 순수 JDBC는 직접 쓰지 않는다.

## 어노테이션(Annotation)이란? (Kotlin이랑 비슷한 이유)

> **질문**: 어노테이션이 뭔가 Kotlin이랑 비슷한데?

### 어노테이션 = 코드에 붙이는 "표식(메타데이터)"

`@Entity`, `@Id`, `@Autowired`, `@Bean`, `@Override`처럼 **`@`로 시작해 클래스·메서드·필드에 붙이는 표식**이다.
**그 자체로 동작을 실행하지는 않고**, 프레임워크(스프링·JPA)가 **읽어서 "이건 이렇게 처리해라"고 판단하는 부가 정보**다.

```java
@Entity                                 // "이 클래스는 DB 테이블과 매핑되는 엔티티야" (JPA가 읽음)
public class Member {
    @Id                                 // "이 필드가 기본키(PK)야"
    @GeneratedValue(...)                // "PK는 자동 생성해"
    private Long id;
}

@Configuration                          // "이건 설정 클래스야" (스프링이 읽음)
public class SpringConfig {
    @Bean                               // "이 메서드 반환값을 스프링 빈으로 등록해"
    public MemberService memberService() { ... }
}
```

- 즉 어노테이션은 **"프레임워크에게 보내는 쪽지"** 같은 것. 스프링/JPA는 시작할 때 이 표식들을 **리플렉션으로 읽어** 빈 등록·테이블 매핑·DI 등을 수행한다.
- 종류:
  - `@Override` → **컴파일러**가 읽음(오버라이드 검증)
  - `@Entity`, `@Autowired`, `@Bean` → **프레임워크**가 런타임에 읽음

### Kotlin과 비슷해 보이는 이유

**Kotlin도 똑같은 어노테이션 문법(`@`)을 쓴다.** 둘 다 **JVM 위에서 도는 언어**라 자바 어노테이션을 그대로 공유하기 때문이다.

```kotlin
// Kotlin — 어노테이션은 사실상 동일하게 생겼다
@Entity
class Member(
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,
    val name: String = ""
)
```

- 그래서 스프링/JPA를 **Java로 쓰든 Kotlin으로 쓰든 `@Entity`, `@Autowired` 등은 똑같이** 붙인다.
- "비슷하다"고 느낀 게 정확하다 — **개념도 문법도 같다.** (Python의 데코레이터 `@`, TypeScript의 데코레이터도 비슷한 발상)

> 한 줄 기억: **어노테이션 = 코드에 붙여 프레임워크에게 의미를 알려주는 표식.**
> Kotlin도 같은 `@` 어노테이션을 쓴다(둘 다 JVM 언어라 공유).

## 용어 정리

| 용어 | 설명 |
| --- | --- |
| 임베디드 모드 | 접속 프로세스가 DB 파일을 직접 여는 방식 (`jdbc:h2:~/test`), 파일 독점 |
| TCP(서버) 모드 | H2 서버가 파일을 열고 클라이언트는 TCP로 접근 (`jdbc:h2:tcp://...`), 동시 접근 가능 |
| 파일 락(file lock) | DB 파일 손상 방지를 위해 H2가 거는 잠금. 임베디드 동시 접근 시 충돌 원인 |
| `DataSource` | DB 커넥션을 얻는 객체. 부트가 자동 생성·빈 등록 |
| `JdbcTemplate` | JDBC 반복 코드를 제거해 주는 스프링 헬퍼 (SQL은 직접) |
| `RowMapper` | ResultSet 한 행을 객체로 변환하는 콜백 |
| JPA / `EntityManager` | 객체-관계 매핑 표준. `persist`/`find`/JPQL로 DB 조작 |
| JPQL | 테이블이 아니라 엔티티(객체)를 대상으로 작성하는 쿼리 |
| `@Transactional` | 메서드 트랜잭션 시작→커밋/롤백 관리. JPA 데이터 변경에 필수 |
| 스프링 데이터 JPA | 인터페이스만으로 CRUD·이름 기반 쿼리·페이징 제공 |
| OCP | 개방-폐쇄 원칙: 확장에 열려 있고 수정에 닫혀 있음 |
